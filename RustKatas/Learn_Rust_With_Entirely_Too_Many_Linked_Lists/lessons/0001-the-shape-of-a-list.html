The Shape of a List

Before we can talk about ownership, we need to see what a list actually is — as boxes of memory the machine can touch. This lesson covers only the first 16 lines of first.rs: the three type definitions. By the end you'll be able to write them from a blank file and explain why they have to look this way.

The one win A linked list is a chain of nodes where each node owns the next one. The whole design of first.rs exists to make that chain representable in memory with a size the compiler can compute.

1 · Two places a value can live: the stack and the heap

Rust is your first language, so let's start from the metal. When your program runs, its data lives in one of two regions of memory:

The Stack

fast, orderedgrows & shrinks as functions are called and return
fixed, known sizethe compiler must know how many bytes at compile time

The Heap

flexiblefor data whose size isn't known up front, or that must outlive one function call
reached by a pointeran address stored on the stack points to the real data on the heap

The stack is like a stack of plates: the machine only ever adds to or removes from the top, so it's extremely fast — but to reserve space, the compiler must know the exact size of every value in advance. That single rule is the villain of this entire lesson. Hold onto it.

Vocabulary A pointer is just a value that holds a memory address — a number that says "the real thing is over there." On a 64-bit machine a pointer is 8 bytes, always, no matter how big the thing it points to is.

2 · struct — a named bundle of fields

A struct glues several values together under one name. Line 13–16 of first.rs:

struct Node {
    elem: i32,
    next: Link,
}
struct Node { … }
struct = "define a new type that bundles fields together." Node is the type's name.
elem: i32
A field named elem of type i32 — a 32-bit signed integer. This is the actual data a list node carries.
next: Link
A field named next of type Link — this is what chains one node to the following one. We define Link next.

So a Node is "an integer, plus a link to whatever comes after it." That's the essence of a linked list: each node knows only about the next one.

3 · enum — a value that is exactly one of several shapes

A list needs to express "there is a next node" or "this is the end." Rust's tool for "one of these possibilities" is the enum. Lines 8–11:

enum Link {
    Empty,
    More(Box<Node>),
}
enum Link { … }
enum = "a value of type Link is exactly one of the following variants at any moment." Never both, never neither.
Empty
The first variant. Carries no data — it just means "the chain ends here." This is how a list represents emptiness without null.
More(Box<Node>)
The second variant. It carries data: a Box<Node>, i.e. "there is another node, and here's how to reach it."
Why no null? In many languages the "end of list" is a special null pointer, and forgetting to check for it crashes programs. Rust makes emptiness a named variant (Empty) that you're forced to handle. The bug class simply can't exist. Ownership's cousin: safety by making the compiler check.

4 · The problem that forces Box into existence

Now the crux. Suppose we'd written the More variant to hold a Node directly, with no Box:

// imaginary, BROKEN version
enum Link { Empty, More(Node) }   // Node contains a Link...
struct Node { elem: i32, next: Link }  // ...which contains a Node...

Ask the compiler its one question: how many bytes is a Node?

  • A Node = an i32 (4 bytes) + a Link.
  • A Link (the More case) = a whole Node.
  • …which contains a Link, which contains a Node, which…

The size is 4 + 4 + 4 + … forever. Infinite. The compiler literally cannot reserve stack space for it, and it rejects the program:

error[E0072]: recursive type `Link` has infinite size

The fix is to break the chain of "contains" with a pointer. That's exactly what Box<Node> is.

5 · Box<Node> — a fixed-size handle to a heap-allocated Node

A Box<Node> is a pointer that owns a Node living on the heap. Two consequences, both essential:

  • Fixed size. A Box is just an address — 8 bytes — no matter how big the Node behind it is. So the compiler's question now has an answer: a Link is at most "a tag + 8 bytes." Finite. Compiles.
  • Owns the heap data. When the Box is dropped, the Node it points to is freed automatically. No manual free, no garbage collector. This is ownership, and it's the thread we'll pull on for the rest of the course.

The List itself is tiny and lives on the stack (lines 1–3):

pub struct List {
    head: Link,
}
pub struct List
pub = "public": code outside this file may use the List type. (Its head field is not pub, so it stays private — an implementation detail.)
head: Link
The list is nothing but a single Link — the front of the chain. Everything else hangs off the heap behind it.

Check yourself

Why does More hold Box<Node> instead of Node?

A Node containing a Node containing a Node… has no computable size, so the compiler rejects it. A Box is a fixed 8-byte pointer, which gives the type a finite size. (Box isn't about speed — heap access is usually slower; and enums can hold structs fine.)

What does the Empty variant replace, compared to older languages?

Empty is a named, must-be-handled way to say "the chain ends here" — the job a null pointer does elsewhere, but without the crash risk, because the compiler forces you to handle it.

Where does the List struct itself live, and where do its nodes live?

The List is just one small Link, so it sits on the stack. Each Box<Node> allocates its Node on the heap and holds the address.
